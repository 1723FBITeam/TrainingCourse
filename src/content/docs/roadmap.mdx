## How to Use This Roadmap

This roadmap is a suggestion, not a requirement. Every team's schedule is different — some start programming in the off-season, others don't begin until kickoff. Adapt this guide to fit your team's timeline, meeting frequency, and competition schedule.

The key idea: work through the units in order, but don't rush. It's better to deeply understand Unit 4 than to skim through all three advanced units.

---

## Pre-Season (Weeks 1–4)

**Focus:** Build your foundation.

- **Weeks 1–2:** Complete the Core Curriculum (Units 0–3). Every programmer on the team should be able to read and trace code in your robot project.
- **Weeks 3–4:** Start Unit 4 lessons. Focus on WPILib deep dive, Java patterns, and understanding constants/subsystems/commands at a deeper level.

:::tip[Team Goal]
By the end of Week 4, every programmer should be able to open any file in the robot project and explain what it does using the Code Reading Framework.
:::

---

## Build Season (Weeks 5–10)

**Focus:** Apply skills to the new robot.

- **Weeks 5–6:** Complete Unit 4 (NetworkTables, Git workflow, reading auto routines). Start contributing safe edits to the new season's code.
- **Weeks 7–8:** Begin Unit 5 (deploy/test cycle, CAN bus debugging, Chief Delphi research, reading top team code, debugging with data). Use these skills as you encounter real problems on the new robot.
- **Weeks 9–10:** Continue Unit 5 (vendor docs, swerve concepts, PathPlanner). These become critical as the robot comes together.

:::tip[Team Goal]
By Week 10, programmers should be able to deploy code, debug hardware issues, and contribute to autonomous routines.
:::

---

## Competition Prep (Weeks 11–14)

**Focus:** Polish and prepare.

- **Weeks 11–12:** Complete Unit 5 (PathPlanner path creation, auto routine building, comparison exercises). Your autonomous routines should be tested and reliable.
- **Weeks 13–14:** Start Unit 6 selectively — focus on competition readiness (Lesson 6.19) and pose estimation (Lesson 6.8) if your team uses vision-based localization.

:::tip[Team Goal]
By your first competition, the programming team should have a pre-match checklist, know how to read match logs, and have a plan for pit debugging.
:::

---

## Competition Season (Weeks 15+)

**Focus:** Debug, iterate, improve.

- Use Unit 6 content as needed based on what problems you encounter at competition.
- **Match log analysis** (Activity 6.9) — review every match and identify what went wrong.
- **Competition debugging** (Activity 6.20) — practice diagnosing issues under time pressure.
- **Hotfix branches** (Lesson 6.19) — make targeted fixes between matches without breaking stable code.

:::caution[Competition Priority]
At competition, reliability beats performance. Don't deploy untested changes. Use the deploy-test-fix cycle from Unit 5.
:::

---

## Off-Season

**Focus:** Level up for next year.

- Complete remaining Unit 6 content: AdvantageKit, simulation, control theory, architecture patterns.
- Work through comparison exercises — study how top teams solved this year's game challenges.
- Experiment with advanced PathPlanner features, PID tuning, and state machines in a low-pressure environment.
- Consider adopting AdvantageKit or other logging frameworks for next season.

:::tip[Off-Season is Where Great Teams Are Made]
The teams that improve the most between seasons are the ones that use the off-season to learn, not just build. Use this time to work through the advanced content without competition pressure.
:::
